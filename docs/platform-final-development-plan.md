# خطة استكمال منصة المئة التعليمية

هذه الوثيقة هي خريطة العمل التنفيذية الحالية للمنصة. الهدف أن يظل التطوير منظمًا، بدون تغيير شكل المنصة أو كسر أي جزء قائم، مع تحويل كل شيء إلى تشغيل حقيقي قابل للتوسع.

## العقد الفني

- الحفاظ على الشكل الحالي والديناميكية الحالية.
- عدم حذف أي فكرة موجودة، بل تفعيلها وربطها وتحسينها.
- MongoDB Atlas هو مصدر البيانات الأساسي في هذه المرحلة.
- GitHub هو مصدر الكود، وVercel للواجهة، وRender للـ API.
- كل محتوى تعليمي يجب أن يرتبط بهذا التسلسل: مسار، مادة، مهارة رئيسية، مهارة فرعية عند الحاجة.
- مركز المهارات هو مصدر المهارات الوحيد، ولا يتم خلطه مع موضوعات التأسيس.
- الموضوعات التأسيسية هي مساحة تعلم ومحتوى، وليست مصدرًا للمهارات.
- أي اختبار يقيس المهارات من الأسئلة المرتبطة به، وليس من حقل مهارات منفصل داخل الاختبار.

## ما تم تنفيذه فعليًا

- تشغيل حقيقي للواجهة والـ API وقاعدة البيانات.
- تسجيل دخول وأدوار أساسية: مدير، معلم، مشرف، طالب، ولي أمر.
- ربط المسارات والمواد والمهارات والأسئلة والدروس والاختبارات والمكتبة ببيانات حقيقية.
- جعل نتائج الاختبارات وتحليل المهارات مبنية على أسئلة الاختبار.
- فصل "اختباراتي" كمحاولات الطالب السابقة، و"اختبارات ساهر" كتجربة اختبار/إنشاء اختبار.
- دعم اختبارات موجهة من الإدارة أو المعلم أو المشرف للمجموعات والطلاب.
- دعم إخفاء وإظهار المحتوى والمسارات وحالات النشر.
- دعم الباقات العامة وباقات المدارس بشكل منفصل.
- إضافة معاينة إدارية للدورات والدروس والاختبارات والمكتبة قبل النشر.
- إضافة مؤشر جاهزية المنصة داخل لوحة المدير قبل النشر.
- تحسين نتيجة الاختبار بصريًا وتبسيط التوصيات.
- تحسين مشغل الفيديو والفشل الآمن عند وجود رابط خارجي.
- دعم محرر رياضيات/عرض KaTeX في أجزاء المنصة.
- دعم استيراد الأسئلة من Excel عبر نموذج منظم.
- دعم الذكاء الاصطناعي عبر طبقة API آمنة، مع قابلية تشغيل Gemini أو Ollama/Gemma أو تعطيل المزود.

## أولويات التطوير القادمة

1. رحلة الطالب

- تثبيت تجربة موحدة لكل اختبار: بدء الاختبار، حل، نتيجة مبسطة، مراجعة، تحليل مهارات، توصيات، إعادة اختبار.
- تحسين صفحة النتيجة لتناسب الابتدائي والمتوسط والثانوي والجوال والتابلت واللاب.
- تحويل صفحة التقارير إلى مدخل رئيسي للمهارات: مهارات ضعيفة، متوسطة، متقنة، ثم اقتراح فيديو/تدريب/اختبار/حصة.
- إضافة تصدير PDF للطالب وولي الأمر والمعلم والمشرف.

2. مساحة التعلم

- تثبيت إدارة مساحة التعلم لكل مادة: دورات، تأسيس، تدريب، اختبارات محاكية، مكتبة.
- كل عنصر داخل مساحة التعلم له: حالة نشر، إخفاء/إظهار، مجاني/مدفوع، معاينة قبل النشر.
- الدورات والموضوعات والتدريبات والاختبارات المحاكية والمكتبة لا تظهر للطالب إلا من إدارة مساحة التعلم، وليس من المستودعات مباشرة.
- المستودعات مثل مركز الدروس ومركز الاختبارات وبنك الأسئلة هي مصادر محتوى، وليست واجهة نشر مباشرة.

3. الإدارة والأدوار

- المدير يراجع وينشر كل محتوى المعلمين.
- المعلم يضيف أسئلة ودروس واختبارات ضمن المسارات/المواد المسندة له.
- المشرف يوجه اختبارات لمجموعته ويتابع ضعف الطلاب.
- مدير المدرسة يرى طلاب مدرسته وتقاريرهم وباقات المدرسة.
- ولي الأمر يرى نتائج أبنائه وتوصيات علاجية مبسطة.

4. المدارس والباقات

- باقات المدارس منفصلة عن باقات الطلاب العامة.
- الطالب التابع لمدرسة يحصل على صلاحياته من المدرسة أو الأكواد، وليس من شراء باقة عامة.
- الباقات العامة تظهر في المسار أو المادة حسب إعداد المدير.
- دعم باقات: اختبارات فقط، تأسيس فقط، كمي كامل، مسار كامل، شاملة.
- إدارة المدارس يجب أن تدعم تعديل اسم المدرسة والفصول بدون حذف وإعادة إنشاء.
- باقات المدارس يجب أن تدعم التنشيط والإيقاف المؤقت بدون حذف، حتى يمكن إيقاف باقة لفترة ثم تشغيلها لاحقًا.
- استيراد طلاب المدارس يجب أن يدعم Excel وCSV مع نموذج تحميل جاهز ومعاينة قبل التنفيذ.
- التوسع القادم في المدارس: استيراد مشرفين وأولياء أمور، وربط الطلاب بأولياء الأمور دفعة واحدة، وتقرير PDF لكل مدرسة أو فصل.

5. الذكاء الاصطناعي

- جعل الذكاء الاصطناعي مساعدًا لا مصدرًا وحيدًا للحكم.
- استخدامه في: اقتراح خطة علاج، شرح نتيجة الطالب، توليد أسئلة مبدئية، تلخيص تقارير، اقتراح محتوى مناسب.
- حماية المفاتيح من الواجهة، وتشغيل المزود من السيرفر فقط.
- دعم Ollama/Gemma محليًا أو على سيرفر منفصل لمن يريد تقليل التكلفة.

6. الجودة والتشغيل

- قبل كل دفعة رفع: TypeScript، Build، Smoke API.
- أي جزء تم التأكد أنه يعمل لا يتم تغييره إلا عند الحاجة وباختبار.
- المحافظة على خريطة المنصة حتى لا تختلط صفحات التعلم مع صفحات المستودعات.

## مؤشر التقدم الحالي

- البنية الأساسية والتشغيل: مكتملة بدرجة عالية.
- الإدارة والمحتوى: متقدمة وتحتاج تثبيت تدفق النشر داخل مساحة التعلم.
- رحلة الطالب: متقدمة وتحتاج مزيدًا من التبسيط في النتائج والتقارير.
- المدارس والأدوار: موجودة وتحتاج تعميق التقارير والصلاحيات.
- الباقات والمدفوعات: موجودة كأساس وتحتاج ربطًا أوضح بواجهة المسارات ومساحة التعلم.
- الذكاء الاصطناعي: الأساس موجود ويحتاج دمجًا أعمق في الرحلات التعليمية.

## مبدأ التسليم

المنصة لا تكون "موقع كورسات فقط"، بل نظام:

تشخيص، تحليل، علاج، إتقان.

كل تطوير قادم يجب أن يخدم هذا المبدأ.

## تحديث تنفيذي - 30 أبريل 2026

- تم تثبيت مؤشر جاهزية المدارس داخل إدارة المدارس وفي كروت المدارس، بحيث يعرف المدير قبل التشغيل هل المدرسة لديها فصول، مشرفون، باقات نشطة، وأكواد دخول صالحة.
- تم التأكد أن تعديل اسم المدرسة، تعديل أسماء الفصول، استيراد طلاب Excel/CSV، وتنشيط/إيقاف باقات المدارس بدون حذف موجودة ضمن نفس شاشة الإدارة.
- نتيجة الاختبار الحالية تعمل بوضعين: ملخص بسيط مناسب للطالب وولي الأمر، وتقرير كامل اختياري لمن يحتاج التفاصيل.
- اختبارات التشغيل بعد الدفعة: TypeScript ناجح، Build ناجح، Smoke API ناجح 50/50 بعد استيقاظ Render.

### الأولوية التالية بعد هذا التحديث

1. تقليل التشتت في تقارير الطالب العامة مع إبراز المهارات أولًا ثم زر واضح للمقترحات: فيديو، تدريب، اختبار، أو حصة.
2. تعميق إدارة الباقات العامة داخل المسارات والمواد وربطها بالمحتوى المفتوح/المغلق.
3. استكمال فحص رحلة مساحة التعلم للطالب على الجوال والتابلت واللاب توب مع بيانات تجريبية مفتوحة ومغلقة.
4. توسيع المدارس لاحقًا باستيراد مشرفين وأولياء أمور وربطهم بالطلاب دفعة واحدة.
